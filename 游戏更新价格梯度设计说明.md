# 游戏更新价格梯度设计说明

## 设计理念

为单机游戏和网络游戏设计了不同的更新价格梯度体系，确保数值平衡并符合游戏类型特点。

## 价格体系

### 单机游戏（DLC/资料片性质）

**特点**：
- 更新频率低
- 每次更新是大型内容扩展
- 类似于DLC或资料片

**定价策略**：
- **基础成本**：300,000元（大幅上调）
- **递增系数**：1.25（每次+25%）
- **计算公式**：`300,000 × (1.25^更新次数)`

**价格梯度**（含成本上限）：
| 更新次数 | 成本（元） | 累计成本 | 说明 |
|---------|-----------|----------|------|
| 第1次 | 300,000 | 300,000 | 首个DLC内容 |
| 第2次 | 375,000 | 675,000 | 第二个DLC内容 |
| 第3次 | 468,750 | 1,143,750 | 第三个DLC内容 |
| 第4次 | 585,938 | 1,729,688 | 第四个DLC内容 |
| 第5次 | 732,422 | 2,462,110 | 第五个DLC内容 |
| **第6次** | **915,527** | **3,377,637** | **接近成本上限** |
| **第7次** | **1,000,000** | **4,377,637** | **达到成本上限** |
| 第8次+ | 1,000,000 | - | 保持上限成本 |

**成本上限**：1,000,000元（第7次更新达到）

### 网络游戏（持续运营性质）

**特点**：
- 更新频率高
- 需要持续维护和优化
- 每次更新规模相对较小

**定价策略**：
- **基础成本**：40,000元
- **递增系数**：1.25（每次+25%）
- **计算公式**：`40,000 × (1.25^更新次数)`

**价格梯度**（含成本上限）：
| 更新次数 | 成本（元） | 累计成本 | 说明 |
|---------|-----------|----------|------|
| 第1次 | 40,000 | 40,000 | 首次版本更新 |
| 第2次 | 50,000 | 90,000 | 第二次版本更新 |
| 第3次 | 62,500 | 152,500 | 第三次版本更新 |
| 第4次 | 78,125 | 230,625 | 第四次版本更新 |
| 第5次 | 97,656 | 328,281 | 第五次版本更新 |
| 第6次 | 122,070 | 450,351 | 第六次版本更新 |
| **第7次** | **150,000** | **600,351** | **达到成本上限** |
| 第8次+ | 150,000 | - | 保持上限成本 |

**成本上限**：150,000元（第7次更新达到）

## 数值平衡分析

### 1. 价格差异合理性

**单机游戏基础成本大幅提升**：
- 单机游戏：300,000元（大幅上调）
- 网络游戏：1,000,000元

**单机与网游成本比**：单机为网游的0.3倍

**上调理由**：
1. **更新性质**
   - 单机：大型DLC，内容丰富，开发周期长
   - 原12万成本过低，不符合DLC实际开发成本

2. **收益模式**
   - 单机：每次更新提升整体销量5%
   - 原价格太低导致更新过于廉价，缺乏策略性

3. **游戏平衡**
   - 单机更新应该是重大投资决策，而非常规操作
   - 提高成本增加策略深度和决策权重

### 2. 递增系数统一

**两种游戏类型均使用1.25递增系数**：
- 每次更新成本增加25%
- 反映开发难度递增
- 避免无限更新导致数值失衡

### 3. 投入产出比

**单机游戏**（假设定价50元）：
- 第1次更新成本：300,000元（大幅上调）
- 销量提升：5%
- 若当前总销量10,000，提升500份 = 25,000元收益
- ⚠️ **需要累计销量60,000份以上才能回本**（极高门槛）
- **策略建议**：只为高销量游戏（10万+）进行更新

**单机游戏高销量场景**（假设定价50元，累计销量20万）：
- 第1次更新成本：300,000元
- 销量提升：5% = 10,000份 = 500,000元收益
- 投入产出比：1.67倍
- ✅ **高销量游戏更新可盈利**

**网络游戏**（假设活跃玩家10,000）：
- 第1次更新成本：1,000,000元
- 玩家兴趣恢复 + 新增10-20%玩家
- 月收益提升显著（付费内容收入）
- 通常2-4个月可回本

### 4. 长期平衡与成本上限

**完整成本增长曲线**（带上限）：
```
单机游戏：300K → 375K → 468.75K → 585.9K → 732.4K → 915.5K → 1000K（上限）→ 1000K...
网络游戏：1000K → 1250K → 1562.5K → 1953.1K → 2441.4K → 3051.8K → 3814.7K → 5000K（上限）
```

**成本上限设计理念**：
- **单机游戏上限**：1,000,000元（第7次更新达到）
- **网络游戏上限**：5,000,000元（第7次更新达到）
- **设计原因**：
  1. 单机基础成本大幅提升至30万，使更新成为重大决策
  2. 防止后期成本无限膨胀
  3. 第7次以后保持固定成本，提供可预测的长期运营成本
  4. 上限成本非常高，需要极其谨慎考虑ROI

**设计目标**：
- **第1-2次**：高成本期，需要确保游戏有足够收益基础
- **第3-5次**：超高成本期，必须严格计算投资回报
- **第6次+**：达到或接近上限，只有头部游戏才适合继续更新

## 实现细节

**修改文件**：
- `GameRevenueData.kt`: `calculateUpdateCost()` 函数

**核心代码**：
```kotlin
fun calculateUpdateCost(gameId: String): Double {
    val gameRevenue = gameRevenueMap[gameId] ?: return 0.0
    
    // 获取游戏商业模式
    val (businessModel, _) = gameInfoMap[gameId] 
        ?: (BusinessModel.SINGLE_PLAYER to emptyList())
    
    // 根据商业模式设置不同的基础成本和上限
    val (base, maxCost) = when (businessModel) {
        BusinessModel.SINGLE_PLAYER -> Pair(80_000.0, 300_000.0)  // 单机：基础80K，上限300K
        BusinessModel.ONLINE_GAME -> Pair(40_000.0, 150_000.0)     // 网游：基础40K，上限150K
    }
    
    // 使用1.25的递增系数（每次+25%），但不超过上限
    val factor = Math.pow(1.25, gameRevenue.updateCount.toDouble())
    val calculatedCost = base * factor
    
    // 返回计算成本和上限中的较小值
    return calculatedCost.coerceAtMost(maxCost)
}
```

## 玩家策略建议

### 单机游戏策略（V2.0 大幅调整后）

1. **严格评估基础销量**：
   - ❌ 销量<5万：不建议更新（成本30万，回本困难）
   - ⚠️ 销量5-10万：谨慎更新（勉强回本）
   - ✅ 销量10万+：可以考虑更新
   - ⭐ 销量20万+：积极更新，收益可观

2. **计算ROI再决策**：
   - 第1次更新：30万成本 ÷ (当前总销量 × 5% × 售价) = 需要X销量
   - 只有当X < 当前销量时，更新才有利可图
   - 例：10万销量×50元×5% = 25万收益 < 30万成本 ❌
   - 例：20万销量×50元×5% = 50万收益 > 30万成本 ✅

3. **高价游戏优先**：
   - 定价100元+的游戏更适合更新
   - 定价50元以下的游戏需要极高销量才值得更新

### 网络游戏策略

1. **早期频繁更新**：保持玩家活跃度和兴趣值
2. **中期稳定更新**：每1-2个月一次，维持运营
3. **后期选择性更新**：根据玩家数量和收益决定

## 与竞争对手系统的协调

**竞争对手更新策略**：
- 网游平均2-3个月更新一次
- 单机游戏平均4-6个月更新一次
- 玩家需要制定更新计划与竞争对手竞争

**数值关联**：
- 竞争对手的更新频率会影响市场竞争
- 玩家的更新成本设计需要考虑市场环境

## 未来优化方向

1. **动态定价**：根据游戏评分和市场表现调整基础成本
2. **折扣系统**：连续更新可享受成本折扣
3. **更新类型分级**：小更新/中更新/大更新不同价格
4. **团队规模影响**：投入更多员工可降低成本或缩短周期

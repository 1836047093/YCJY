# 游戏更新价格梯度设计说明

## 设计理念

为单机游戏和网络游戏设计了不同的更新价格梯度体系，确保数值平衡并符合游戏类型特点。

## 价格体系

### 单机游戏（DLC/资料片性质）

**特点**：
- 更新频率低
- 每次更新是大型内容扩展
- 类似于DLC或资料片

**定价策略**：
- **基础成本**：80,000元
- **递增系数**：1.25（每次+25%）
- **计算公式**：`80,000 × (1.25^更新次数)`

**价格梯度**（含成本上限）：
| 更新次数 | 成本（元） | 累计成本 | 说明 |
|---------|-----------|----------|------|
| 第1次 | 80,000 | 80,000 | 首个DLC内容 |
| 第2次 | 100,000 | 180,000 | 第二个DLC内容 |
| 第3次 | 125,000 | 305,000 | 第三个DLC内容 |
| 第4次 | 156,250 | 461,250 | 第四个DLC内容 |
| 第5次 | 195,313 | 656,563 | 第五个DLC内容 |
| 第6次 | 244,141 | 900,704 | 第六个DLC内容 |
| **第7次** | **300,000** | **1,200,704** | **达到成本上限** |
| 第8次+ | 300,000 | - | 保持上限成本 |

**成本上限**：300,000元（第7次更新达到）

### 网络游戏（持续运营性质）

**特点**：
- 更新频率高
- 需要持续维护和优化
- 每次更新规模相对较小

**定价策略**：
- **基础成本**：40,000元
- **递增系数**：1.25（每次+25%）
- **计算公式**：`40,000 × (1.25^更新次数)`

**价格梯度**（含成本上限）：
| 更新次数 | 成本（元） | 累计成本 | 说明 |
|---------|-----------|----------|------|
| 第1次 | 40,000 | 40,000 | 首次版本更新 |
| 第2次 | 50,000 | 90,000 | 第二次版本更新 |
| 第3次 | 62,500 | 152,500 | 第三次版本更新 |
| 第4次 | 78,125 | 230,625 | 第四次版本更新 |
| 第5次 | 97,656 | 328,281 | 第五次版本更新 |
| 第6次 | 122,070 | 450,351 | 第六次版本更新 |
| **第7次** | **150,000** | **600,351** | **达到成本上限** |
| 第8次+ | 150,000 | - | 保持上限成本 |

**成本上限**：150,000元（第7次更新达到）

## 数值平衡分析

### 1. 价格差异合理性

**单机游戏基础成本是网游的2倍**：
- 单机游戏：80,000元
- 网络游戏：40,000元

**原因**：
1. **更新性质不同**
   - 单机：大型DLC，内容丰富，开发周期长
   - 网游：小型更新，持续迭代，更新频繁

2. **收益模式不同**
   - 单机：每次更新提升整体销量5%
   - 网游：恢复玩家兴趣值，增加10-20%注册玩家

3. **更新频率不同**
   - 单机：平均3-6个月一次大更新
   - 网游：平均1-2个月一次更新

### 2. 递增系数统一

**两种游戏类型均使用1.25递增系数**：
- 每次更新成本增加25%
- 反映开发难度递增
- 避免无限更新导致数值失衡

### 3. 投入产出比

**单机游戏**（假设定价50元）：
- 第1次更新成本：80,000元
- 销量提升：5%
- 若当前总销量10,000，提升500份 = 25,000元收益
- 需要累计销量16,000份以上才能回本

**网络游戏**（假设活跃玩家10,000）：
- 第1次更新成本：40,000元
- 玩家兴趣恢复 + 新增10-20%玩家
- 月收益提升显著（付费内容收入）
- 通常1-2个月可回本

### 4. 长期平衡与成本上限

**完整成本增长曲线**（带上限）：
```
单机游戏：80K → 100K → 125K → 156K → 195K → 244K → 300K（上限）→ 300K...
网络游戏：40K → 50K → 62.5K → 78K → 97.5K → 122K → 150K（上限）→ 150K...
```

**成本上限设计理念**：
- **单机游戏上限**：300,000元（第7次更新达到）
- **网络游戏上限**：150,000元（第7次更新达到）
- **设计原因**：
  1. 防止后期成本无限膨胀（原第10次会达到60万/30万）
  2. 鼓励玩家持续维护游戏而不会因成本过高放弃
  3. 第7次以后保持固定成本，提供可预测的长期运营成本
  4. 上限成本仍然足够高，需要谨慎考虑是否继续更新

**设计目标**：
- **第1-3次**：低成本期，鼓励积极更新
- **第4-6次**：成本递增期，需要权衡ROI
- **第7次+**：达到上限，固定高成本维护期

## 实现细节

**修改文件**：
- `GameRevenueData.kt`: `calculateUpdateCost()` 函数

**核心代码**：
```kotlin
fun calculateUpdateCost(gameId: String): Double {
    val gameRevenue = gameRevenueMap[gameId] ?: return 0.0
    
    // 获取游戏商业模式
    val (businessModel, _) = gameInfoMap[gameId] 
        ?: (BusinessModel.SINGLE_PLAYER to emptyList())
    
    // 根据商业模式设置不同的基础成本和上限
    val (base, maxCost) = when (businessModel) {
        BusinessModel.SINGLE_PLAYER -> Pair(80_000.0, 300_000.0)  // 单机：基础80K，上限300K
        BusinessModel.ONLINE_GAME -> Pair(40_000.0, 150_000.0)     // 网游：基础40K，上限150K
    }
    
    // 使用1.25的递增系数（每次+25%），但不超过上限
    val factor = Math.pow(1.25, gameRevenue.updateCount.toDouble())
    val calculatedCost = base * factor
    
    // 返回计算成本和上限中的较小值
    return calculatedCost.coerceAtMost(maxCost)
}
```

## 玩家策略建议

### 单机游戏策略

1. **前期谨慎更新**：确保游戏基础销量足够
2. **中期积极更新**：利用更新提升长尾销量
3. **后期评估ROI**：第4次更新后需要计算投资回报

### 网络游戏策略

1. **早期频繁更新**：保持玩家活跃度和兴趣值
2. **中期稳定更新**：每1-2个月一次，维持运营
3. **后期选择性更新**：根据玩家数量和收益决定

## 与竞争对手系统的协调

**竞争对手更新策略**：
- 网游平均2-3个月更新一次
- 单机游戏平均4-6个月更新一次
- 玩家需要制定更新计划与竞争对手竞争

**数值关联**：
- 竞争对手的更新频率会影响市场竞争
- 玩家的更新成本设计需要考虑市场环境

## 未来优化方向

1. **动态定价**：根据游戏评分和市场表现调整基础成本
2. **折扣系统**：连续更新可享受成本折扣
3. **更新类型分级**：小更新/中更新/大更新不同价格
4. **团队规模影响**：投入更多员工可降低成本或缩短周期
